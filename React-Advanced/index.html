1. TypeScript in React – Typen für mehr Sicherheit

Bisher hast du JavaScript geschrieben. Das Problem: JavaScript prüft nicht, welche Art von Daten du übergibst. Du kannst aus Versehen eine Zahl übergeben, wo ein Text erwartet wird – und der Fehler fällt erst im Browser auf.

TypeScript ist JavaScript mit einem Extra: Es prüft deine Datentypen bevor der Code läuft. Stell dir vor, JavaScript ist wie ein offener Briefkasten – jeder kann alles reinwerfen. TypeScript ist wie ein Briefkasten mit einem Schlitz in einer bestimmten Form – nur das Richtige passt rein.

Die wichtigsten Typen:

// Einfache Typen
let name: string = "Anna";       // Text
let age: number = 25;             // Zahl
let isActive: boolean = true;     // Wahr/Falsch

// Arrays (Listen)
let hobbies: string[] = ["Lesen", "Kochen", "Sport"];
let scores: number[] = [95, 87, 100];

Für Objekte (also Daten mit mehreren Eigenschaften) brauchst du einen Bauplan, der beschreibt, welche Felder ein Objekt haben muss und welchen Typ jedes Feld hat. Dafür gibt es in TypeScript zwei Wege: interface und type. 

// Weg 1: interface – der Bauplan für ein Objekt
interface User {
  name: string;
  age: number;
  email: string;
  isAdmin: boolean;
}

// Weg 2: type – kann das Gleiche
type User = {
  name: string;
  age: number;
  email: string;
  isAdmin: boolean;
}

// Egal ob interface oder type – die Nutzung ist identisch:
let user: User = {
  name: "Anna",
  age: 25,
  email: "anna@example.com",
  isAdmin: false
};

Beide Varianten funktionieren für Objekte fast gleich. Der Unterschied: type kann auch für andere Dinge genutzt werden (z.B. Union Types wie type Status = "active" | "inactive"), während interface speziell für die Form von Objekten gedacht ist.

In den Folgenden Beispielen nutzen wir interface. Das hat aber keinen technischen Grund. Die Nutzung von type ist genauso richtig und an manchen Stellen benötigt, bspw. bei Union-Types von Reducern

TypeScript bei Props – so typisierst du Komponenten:

// Interface für die Props definieren
interface ProfileCardProps {
  name: string;
  age: number;
  city: string;
}

// Props mit dem Interface typisieren
function ProfileCard({ name, age, city }: ProfileCardProps) {
  return (
    <div>
      <h2>{name}</h2>
      <p>Alter: {age}</p>
      <p>Stadt: {city}</p>
    </div>
  );
}

// TypeScript warnt, wenn du etwas Falsches übergibst:
<ProfileCard name="Anna" age={25} city="Berlin" />  // OK
<ProfileCard name="Anna" age="25" city="Berlin" />  // FEHLER! age muss eine Zahl sein

Falsch:

// Ohne Typen: Kein Fehler beim Schreiben, Absturz im Browser
function ProfileCard({ name, age, city }) {
  return <p>{name} ist {age.toFixed(0)} Jahre alt</p>;
}

// age ist aus Versehen ein String:
<ProfileCard name="Anna" age="fünfundzwanzig" city="Berlin" />
// Absturz: "fünfundzwanzig".toFixed is not a function

Richtig:

// Mit Typen: TypeScript erkennt den Fehler SOFORT
interface ProfileCardProps {
  name: string;
  age: number;    // <-- muss eine Zahl sein
  city: string;
}

function ProfileCard({ name, age, city }: ProfileCardProps) {
  return <p>{name} ist {age.toFixed(0)} Jahre alt</p>;
}

// TypeScript zeigt sofort einen roten Fehler:
// "Type 'string' is not assignable to type 'number'"

useState mit TypeScript:

// TypeScript erkennt den Typ oft automatisch:
const [count, setCount] = useState(0);        // count ist automatisch 'number'
const [name, setName] = useState("Anna");     // name ist automatisch 'string'

// Bei komplexen Typen gibst du ihn explizit an:
interface Todo {
  id: number;
  text: string;
  done: boolean;
}

const [todos, setTodos] = useState<Todo[]>([]);  // Array von Todo-Objekten
const [user, setUser] = useState<User | null>(null);  // User oder null
Merke: TypeScript-Dateien enden auf .tsx (statt .jsx) bzw. .ts (statt .js). Interfaces definieren die Form deiner Daten. Bei useState kannst du den Typ in spitzen Klammern angeben: useState<Typ>(startwert).

2. Vite – dein schnelles Build-Tool

Was Vite erstellt – die Standardstruktur:

mein-projekt/
├── public/ ← Statische Dateien (Bilder etc.)
├── src/ ← Dein gesamter Code lebt hier
│   ├── App.tsx ← Deine Hauptkomponente
│   ├── App.css ← Styling für App
│   ├── main.tsx ← Einstiegspunkt (render)
│   └── index.css ← Globale Styles
├── index.html ← Die einzige HTML-Datei
├── package.json ← Projektinfos & Abhängigkeiten
└── tsconfig.json ← TypeScript-Konfiguration

Die wichtigste Datei ist main.tsx – sie rendert deine App-Komponente in den Browser: 

// main.tsx – der Einstiegspunkt
import React from "react";
import ReactDOM from "react-dom/client";
import App from "./App";
import "./index.css";

ReactDOM.createRoot(document.getElementById("root")!).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
);
Merke: Vite erstellt mit --template react-ts direkt ein TypeScript-Projekt. Der Befehl npm run dev startet den Entwicklungsserver, npm run build erstellt die fertige Website für die Veröffentlichung.

3. Ordnerstruktur – Ordnung im Projekt

Wenn dein Projekt wächst, brauchst du eine klare Ordnerstruktur. Ohne Ordnung findest du nichts mehr wieder – wie ein Kleiderschrank, in dem alles auf einem Haufen liegt.

Die Grundidee: Jede Datei hat ihren Platz. Wir trennen zwischen Seiten (Views), wiederverwendbaren Bausteinen (Components) und Logik (Hooks, Context, Types).

Empfohlene Struktur:

src/
├── components/ ← Wiederverwendbare UI-Bausteine
│   ├── Button/
│   │   ├── Button.tsx
│   │   └── Button.css
│   ├── Header/
│   │   ├── Header.tsx
│   │   └── Header.css
│   └── Card/
│       ├── Card.tsx
│       └── Card.css
├── views/ ← Seitenansichten (eine pro Route)
│   ├── Home/
│   │   ├── Home.tsx
│   │   ├── Home.css
│   │   └── components/ ← Subkomponenten nur für Home
│   │        └── HeroSection.tsx
│   ├── Profile/
│   │   └── Profile.tsx
│   └── Settings/
│       └── Settings.tsx
├── hooks/ ← Eigene Custom Hooks
│   └── useLocalStorage.ts
├── context/ ← Context Provider
│   └── ThemeContext.tsx
├── types/ ← Gemeinsame TypeScript-Typen
│   └── index.ts
├── App.tsx ← Hauptkomponente mit Router
└── main.tsx ← Einstiegspunkt

Die wichtigsten Regeln:

  • components/ – Bausteine, die auf mehreren Seiten verwendet werden (Button, Header, Card). Jede Komponente bekommt einen eigenen Ordner mit .tsx und .css.
  • views/ – Eine View pro Seite/Route. Wenn eine View eine Unterkomponente hat, die nur dort benutzt wird, kommt sie in einen components/-Unterordner innerhalb der View.
  • hooks/ – Eigene Hooks (Funktionen, die mit use beginnen) gehören hierher.
  • context/ – Globaler State mit useContext wird hier definiert.
  • types/ – TypeScript-Interfaces, die in mehreren Dateien gebraucht werden.
Merke: Die Faustregel ist einfach – wird eine Komponente an mehreren Stellen genutzt, kommt sie in components/. Wird sie nur auf einer Seite genutzt, kommt sie in den components/-Unterordner dieser View. Jeder Ordner hat einen klaren Zweck.

4. Wichtige Hooks – useEffect, useRef, useMemo & Custom Hooks

Du kennst schon useState. Jetzt lernst du vier weitere Werkzeuge kennen, die dir in fast jedem Projekt begegnen werden.

useEffect – Seiteneffekte ausführen

useEffect ist wie ein Auftrag, den du React gibst: „Wenn sich etwas ändert, führe diesen Code aus.“ Das brauchst du z.B. für:

  • Daten von einer API laden
  • Den Seiten-Titel ändern
  • Einen Timer starten
  • Auf Änderungen reagieren
import { useState, useEffect } from "react";

function UserProfile({ userId }: { userId: number }) {
  const [user, setUser] = useState<User | null>(null);

  // Wird ausgeführt wenn die Komponente erscheint
  // UND wenn sich userId ändert
  useEffect(function () {
    console.log("Lade Daten für User", userId);

    fetch("/api/users/" + userId)
      .then(function (response) { return response.json(); })
      .then(function (data) { setUser(data); });
  }, [userId]);
  //  ^^^^^^^^ Abhängigkeits-Array: Nur neu ausführen wenn userId sich ändert

  if (user === null) {
    return <p>Laden...</p>;
  }

  return <h1>{user.name}</h1>;
}
useEffect Funktion + [userId] Abhängigkeit = Ausführen bei Änderung

Das Array am Ende bestimmt, wann der Effekt ausgeführt wird.

Das Abhängigkeits-Array (der zweite Parameter) ist entscheidend: 

// Variante 1: Leeres Array – nur EINMAL beim ersten Anzeigen
useEffect(function () {
  console.log("Komponente wurde geladen");
}, []);

// Variante 2: Mit Werten – bei JEDER Änderung dieser Werte
useEffect(function () {
  console.log("Name oder Alter hat sich geändert");
}, [name, age]);

// Variante 3: KEIN Array – bei JEDEM Rendern (selten nötig!)
useEffect(function () {
  console.log("Komponente wurde neu gerendert");
});
Achtung: Vergiss nie das Abhängigkeits-Array! Ohne Array wird der Effekt bei jedem Rendern ausgeführt – das kann zu Endlosschleifen führen, besonders wenn du im Effekt setState aufrufst.

useRef – Auf HTML-Elemente zugreifen

useRef erstellt eine Referenz – wie ein Zeigefinger, der auf ein bestimmtes Element auf der Seite zeigt. Damit kannst du z.B. ein Eingabefeld fokussieren oder die Höhe eines Elements auslesen. 

import { useRef } from "react";

function SearchBar() {
  const inputRef = useRef<HTMLInputElement>(null);
  //                      ^^^^^^^^^^^^^^^^ Typ des Elements

  function handleButtonClick() {
    // Direkt auf das Input-Element zugreifen:
    inputRef.current?.focus();
  }

  return (
    <div>
      <input ref={inputRef} type="text" placeholder="Suchen..." />
      <button onClick={handleButtonClick}>Fokussieren</button>
    </div>
  );
}

useMemo – Berechnungen zwischenspeichern

useMemo merkt sich das Ergebnis einer Berechnung und führt sie nur dann erneut aus, wenn sich die Eingangswerte ändern. Das ist wie ein Taschenrechner mit Speicher – wenn du die gleiche Aufgabe nochmal eingibst, zeigt er sofort das gespeicherte Ergebnis. 

import { useState, useMemo } from "react";

function ProductList({ products }: { products: Product[] }) {
  const [filter, setFilter] = useState("");

  // Wird nur neu berechnet wenn products oder filter sich ändern
  const filteredProducts = useMemo(function () {
    console.log("Filtere Produkte...");
    return products.filter(function (p) {
      return p.name.toLowerCase().includes(filter.toLowerCase());
    });
  }, [products, filter]);

  return (
    <div>
      <input value={filter} onChange={function (e) { setFilter(e.target.value); }} />
      <p>{filteredProducts.length} Produkte gefunden</p>
    </div>
  );
}

Custom Hooks – Eigene Logik auslagern

Wenn du die gleiche Logik in mehreren Komponenten brauchst, kannst du einen eigenen Hook erstellen. Die Regel: Der Name muss mit use beginnen. 

// hooks/useFormInput.ts
import { useState } from "react";

// Dieser Hook bündelt alles, was ein Eingabefeld braucht:
// den aktuellen Wert, eine onChange-Funktion und eine Reset-Funktion.
function useFormInput(initialValue: string) {
  const [value, setValue] = useState(initialValue);

  // Diese Funktion wird direkt an onChange übergeben
  function handleChange(event: React.ChangeEvent<HTMLInputElement>) {
    setValue(event.target.value);
  }

  // Setzt das Feld auf den Startwert zurück
  function reset() {
    setValue(initialValue);
  }

  return { value, onChange: handleChange, reset };
}

export default useFormInput;

// Nutzung in einer Komponente:
function ContactForm() {
  const nameInput = useFormInput("");
  const emailInput = useFormInput("");

  function handleSubmit() {
    console.log("Name:", nameInput.value, "Email:", emailInput.value);
    nameInput.reset();   // Felder leeren
    emailInput.reset();
  }

  return (
    <div>
      <input type="text" placeholder="Name" value={nameInput.value} onChange={nameInput.onChange} />
      <input type="email" placeholder="Email" value={emailInput.value} onChange={emailInput.onChange} />
      <button onClick={handleSubmit}>Absenden</button>
    </div>
  );
}
Merke: useEffect führt Code nach dem Rendern aus (Daten laden, Timer etc.). useRef zeigt direkt auf ein HTML-Element. useMemo speichert Berechnungsergebnisse. Custom Hooks bündeln wiederverwendbare Logik in einer Funktion, die mit use beginnt.

5. React Router – Mehrere Seiten in einer App

Eine normale Website hat mehrere Seiten (/home, /about, /profile). Jede URL zeigt andere Inhalte. React Router macht das möglich – ohne die Seite jedes Mal komplett neu zu laden.

Stell dir einen Bilderrahmen vor: Der Rahmen (die Navigation, der Header, der Footer) bleibt immer gleich. Nur das Bild im Rahmen wechselt – je nachdem, welche URL du aufrufst.

Installation:

npm install react-router-dom

Grundlegende Einrichtung:

// App.tsx
import { createBrowserRouter, RouterProvider } from "react-router-dom";
import Home from "./views/Home/Home";
import About from "./views/About/About";
import Profile from "./views/Profile/Profile";

// Router wird außerhalb der Komponente erstellt
const router = createBrowserRouter([
  { path: "/", element: <Home /> },
  { path: "/about", element: <About /> },
  { path: "/profile", element: <Profile /> },
]);

function App() {
  return <RouterProvider router={router} />;
}

Navigation mit Link:

import { Link } from "react-router-dom";

function Navigation() {
  return (
    <nav>
      <Link to="/">Home</Link>
      <Link to="/about">Über uns</Link>
      <Link to="/profile">Profil</Link>
    </nav>
  );
}
Achtung: Nutze immer <Link> statt <a href="...">! Ein normaler Link lädt die ganze Seite neu. <Link> wechselt nur den Inhalt – viel schneller!

Parent/Child Routes & Outlet:

Oft möchtest du ein Layout (Header, Footer, Seitenleiste), das auf jeder Seite gleich bleibt. Dafür gibt es verschachtelte Routen: 

// Layout-Komponente mit Outlet
import { Outlet, Link } from "react-router-dom";

function Layout() {
  return (
    <div>
      <header>
        <nav>
          <Link to="/">Home</Link>
          <Link to="/about">Über uns</Link>
        </nav>
      </header>

      <main>
        <Outlet />  {/* Hier wird die aktuelle Seite eingesetzt */}
      </main>

      <footer>
        <p>Meine App © 2025</p>
      </footer>
    </div>
  );
}

// Verschachtelte Routen in App.tsx
const router = createBrowserRouter([
  {
    path: "/",
    element: <Layout />,        // Parent
    children: [
      { index: true, element: <Home /> },          // Index-Route: "/"
      { path: "about", element: <About /> },        // Child: "/about"
      { path: "profile", element: <Profile /> },    // Child: "/profile"
    ],
  },
]);

function App() {
  return <RouterProvider router={router} />;
}
Parent: Layout <Route path="/" element={<Layout />}>
Index-Route <Route index element={<Home />} /> → URL: /
Child-Route <Route path="about" element={<About />} /> → URL: /about
Child-Route <Route path="profile" element={<Profile />} /> → URL: /profile

Dynamische Routen – Variable URLs:

Manchmal ist ein Teil der URL variabel – z.B. /user/1, /user/2, /user/42. Dafür nutzt du einen Doppelpunkt als Platzhalter: 

// Route mit Parameter:
<Route path="/user/:userId" element={<UserDetail />} />

// In der Komponente den Parameter auslesen:
import { useParams } from "react-router-dom";

function UserDetail() {
  const { userId } = useParams();
  // Bei URL /user/42 ist userId = "42"

  return <h1>Profil von User {userId}</h1>;
}
Merke: <Outlet /> ist der Platzhalter im Layout, wo die Kind-Seite eingesetzt wird. index markiert die Standard-Route eines Parents. :parameter in der URL macht sie dynamisch – useParams() liest den Wert aus.

6. State Management – useContext & useReducer

Du kennst useState für State in einer einzelnen Komponente. Aber was, wenn mehrere Komponenten auf die gleichen Daten zugreifen müssen? Oder wenn dein State komplex wird?

Das Problem: Prop Drilling

Ohne Lösung musst du Daten durch viele Ebenen weitergeben – das nennt man Prop Drilling (wie ein Bohrer durch Schichten). Das wird schnell unübersichtlich: 

// Prop Drilling: App → Layout → Header → UserInfo
// Jede Komponente muss user weiterreichen, obwohl nur UserInfo ihn braucht!
function App() {
  const [user, setUser] = useState(currentUser);
  return <Layout user={user} />;
}
function Layout({ user }) {
  return <Header user={user} />;
}
function Header({ user }) {
  return <UserInfo user={user} />;
}

useContext – Daten global verfügbar machen

useContext ist wie ein Schwarzes Brett im Büro: Du hängst Informationen dran, und jeder kann sie lesen – ohne dass sie von Person zu Person weitergereicht werden. 

// 1. Context erstellen (context/ThemeContext.tsx)
import { createContext } from "react";

// Context erstellen mit einem Standardwert
export const ThemeContext = createContext<{
  theme: string;
  toggleTheme: () => void;
}>({
  theme: "hell",
  toggleTheme: function () {},  // Leere Funktion als Platzhalter
});
// 2. Provider direkt in App.tsx einbinden
import { useState } from "react";
import { ThemeContext } from "./context/ThemeContext";

function App() {
  const [theme, setTheme] = useState("hell");

  function toggleTheme() {
    if (theme === "hell") {
      setTheme("dunkel");
    } else {
      setTheme("hell");
    }
  }

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      <RouterProvider router={router} />
    </ThemeContext.Provider>
  );
}

// 3. In JEDER Komponente darauf zugreifen – ohne Prop Drilling!
import { useContext } from "react";
import { ThemeContext } from "../context/ThemeContext";

function ThemeButton() {
  const { theme, toggleTheme } = useContext(ThemeContext);

  return (
    <button onClick={toggleTheme}>
      Aktuell: {theme} – Klick zum Wechseln!
    </button>
  );
}
ThemeContext.Provider stellt Daten bereit → useContext() greift darauf zu → {theme} im JSX

Der Provider umschließt die App. Jede Komponente darunter kann mit useContext(ThemeContext) auf die Daten zugreifen – egal wie tief verschachtelt.

useReducer – Komplexen State verwalten

useReducer ist wie useState, aber für kompliziertere Fälle. Stell dir eine Poststelle vor: Du schickst einen Auftrag (Action) ab, und die Poststelle (der Reducer) entscheidet, was damit passiert. 

import { useReducer } from "react";

// 1. Typen definieren
interface State {
  count: number;
}

type Action =
  | { type: "INCREMENT" }
  | { type: "DECREMENT" }
  | { type: "RESET" };

// 2. Reducer-Funktion: Bekommt alten State + Action, gibt neuen State zurück
function counterReducer(state: State, action: Action): State {
  if (action.type === "INCREMENT") {
    return { count: state.count + 1 };
  }
  if (action.type === "DECREMENT") {
    return { count: state.count - 1 };
  }
  if (action.type === "RESET") {
    return { count: 0 };
  }
  return state;
}

// 3. In der Komponente verwenden
function Counter() {
  const [state, dispatch] = useReducer(counterReducer, { count: 0 });
  //     ^^^^^  ^^^^^^^^                                 ^^^^^^^^^^
  //     State  Funktion zum Senden                      Startwert

  return (
    <div>
      <p>Zähler: {state.count}</p>
      <button onClick={function () { dispatch({ type: "INCREMENT" }); }}>+1</button>
      <button onClick={function () { dispatch({ type: "DECREMENT" }); }}>-1</button>
      <button onClick={function () { dispatch({ type: "RESET" }); }}>Reset</button>
    </div>
  );
}
dispatch { type: "INCREMENT" } → Reducer berechnet neuen State → state { count: 1 }

dispatch sendet eine Action an den Reducer. Der Reducer gibt den neuen State zurück.

Merke: useContext macht Daten global verfügbar (kein Prop Drilling mehr). useReducer eignet sich für komplexen State mit vielen verschiedenen Aktionen. Oft werden beide kombiniert: Context stellt Daten bereit, Reducer verwaltet die Logik.

7. Externe Packages – Axios & Material UI

Du musst nicht alles selbst bauen! Die JavaScript-Welt hat tausende Pakete (Packages), die du mit npm install installieren kannst. Zwei besonders nützliche:

Axios – Daten von APIs laden

Axios ist eine Bibliothek für HTTP-Anfragen (Daten aus dem Internet holen). Du kannst auch fetch benutzen, aber Axios macht einiges einfacher: automatische JSON-Umwandlung, bessere Fehlerbehandlung und die Möglichkeit, Standardwerte zu setzen. 

# Installation:
npm install axios
import axios from "axios";
import { useState, useEffect } from "react";

interface Post {
  id: number;
  title: string;
  body: string;
}

function PostList() {
  const [posts, setPosts] = useState<Post[]>([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState("");

  useEffect(function () {
    axios.get<Post[]>("https://jsonplaceholder.typicode.com/posts")
      .then(function (response) {
        setPosts(response.data);  // Axios packt die Daten automatisch in .data
        setLoading(false);
      })
      .catch(function (err) {
        setError("Fehler beim Laden: " + err.message);
        setLoading(false);
      });
  }, []);

  if (loading) return <p>Laden...</p>;
  if (error) return <p>{error}</p>;

  return (
    <ul>
      {posts.map(function (post) {
        return <li key={post.id}>{post.title}</li>;
      })}
    </ul>
  );
}
useEffect beim Laden → axios.get() API-Anfrage → setPosts() Daten speichern → posts.map() Liste anzeigen

Typisches Muster: Daten laden, in State speichern, im JSX anzeigen.

Material UI (MUI) – Fertige Komponenten

MUI ist eine Sammlung von fertigen, schönen Komponenten – Buttons, Textfelder, Karten, Dialoge und vieles mehr. Statt alles selbst zu stylen, importierst du einfach die passende Komponente. 

# Installation:
npm install @mui/material @emotion/react @emotion/styled
import { Button, TextField, Card, CardContent, Typography } from "@mui/material";
import { useState } from "react";

function ContactForm() {
  const [name, setName] = useState("");

  return (
    <Card sx={{ maxWidth: 400, padding: 2 }}>
      <CardContent>
        <Typography variant="h5">Kontaktformular</Typography>
        <TextField
          label="Dein Name"
          value={name}
          onChange={function (e) { setName(e.target.value); }}
          fullWidth
          margin="normal"
        />
        <Button variant="contained" color="primary">
          Absenden
        </Button>
      </CardContent>
    </Card>
  );
}

Die wichtigsten MUI-Komponenten für den Anfang:

  • Button – Klickbare Buttons (variant: „contained“, „outlined“, „text“)
  • TextField – Eingabefelder mit Label und Validierung
  • Card / CardContent – Karten-Container
  • Typography – Texte mit verschiedenen Größen (h1–h6, body1, body2)
  • IconButton – Buttons mit Icons
  • List / ListItem – Auflistungen
Merke: npm install paketname installiert ein externes Paket. Axios vereinfacht HTTP-Anfragen (Daten laden). MUI liefert fertige Komponenten, damit du dich auf die Logik konzentrieren kannst statt auf das Styling. Das sx-Prop bei MUI erlaubt Inline-Styling als Objekt.
